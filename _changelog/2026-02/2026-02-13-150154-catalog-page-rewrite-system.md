# Catalog Page Rewrite System: From Research Documents to Developer-Facing Documentation

**Date**: February 13, 2026
**Type**: Feature
**Components**: Documentation, Build System, Cursor Rules

## Summary

Replaced the auto-generated research documents in the Planton docs catalog with hand-written, source-verified catalog pages co-located with each component. Established the catalog page standard, wrote Cursor rules for creation and audit, modified the build pipeline for gradual migration, and wrote 5 exemplar catalog pages spanning different providers and complexity levels.

## Problem Statement / Motivation

The docs site catalog at `/docs/catalog/{provider}/{component}` was generated by copying `v1/docs/README.md` — research documents produced during the component forge process. These documents are 300-1000+ lines covering deployment maturity spectrums, tool comparisons, and architectural theory. A developer looking for "how do I deploy an AWS ALB with Planton" got a 400-line essay about load balancer history instead of a manifest they could copy and a field reference they could scan.

### Pain Points

- Catalog pages read like academic papers, not developer documentation
- No manifest examples in the Quick Start position — buried deep if present at all
- No structured configuration reference (field tables with types, defaults, validation)
- Technology comparison sections (Terraform vs CloudFormation vs CDK vs Pulumi) that are irrelevant to the catalog use case
- "What's Not Yet Implemented" sections that undermine confidence
- Inconsistent structure across 178 components

## Solution / What's New

### Catalog Page Standard

Defined a fixed 9-section structure that every `catalog-page.md` must follow:

1. H1 title + overview paragraph
2. What Gets Created (concrete cloud resources)
3. Prerequisites
4. Quick Start (minimal manifest + deploy command)
5. Configuration Reference (Required + Optional field tables)
6. Examples (3-5 progressive manifests)
7. Stack Outputs
8. Related Components

### Co-located Source Files

Each component now has (or will have) a `catalog-page.md` at:
```
apis/dev/planton/provider/{provider}/{component}/v1/catalog-page.md
```

This file is committed as source, hand-written by AI agents following strict Cursor rules, and verified against spec.proto, stack_outputs.proto, and the Pulumi module.

### Gradual Migration

The build script (`copy-component-docs.ts`) now prefers `catalog-page.md` over `docs/README.md`, with automatic fallback. Components with the new catalog pages get the improved experience immediately; the remaining ~173 components continue showing the legacy research documents until rewritten.

## Implementation Details

### Build Script Change

Updated `scanProvider()` in `site/scripts/copy-component-docs.ts` to check for `catalog-page.md` before falling back to `docs/README.md`. Added source type logging (`catalog-page` vs `legacy`) to track migration progress.

### Cursor Rules

Created two rules in `_rules/docs/`:

- **write-planton-component-catalog-page.mdc** — Comprehensive guide for writing new catalog pages. Includes the documentation writer persona, section-by-section content guidelines, source code reading order, formatting rules, anti-patterns, and the 6-point verification protocol.
- **audit-planton-component-catalog-page.mdc** — 8-step audit procedure for verifying existing catalog pages. Covers structural checks, field verification against proto definitions, manifest validation, tone auditing, boundary checks, and formatting consistency.

### 5 Exemplar Catalog Pages

Written and source-verified:

| Component | Provider | Complexity | Key Features |
|-----------|----------|------------|-------------|
| AwsAlb | AWS | Medium | SSL, DNS, foreign keys, multi-AZ validation |
| KubernetesPostgres | Kubernetes | High | Zalando operator, databases, users, backup/restore, ingress |
| GcpGkeCluster | GCP | High | Private cluster, VPC-native, Workload Identity, release channels |
| AwsS3Bucket | AWS | Low-Medium | Encryption, versioning, lifecycle rules, CORS, logging |
| CloudflareWorker | Cloudflare | Medium | R2 script bundles, KV bindings, DNS routing, env/secrets |

Each page was verified against:
- `spec.proto` for field names, types, validation rules, and defaults
- `stack_outputs.proto` for output fields
- `iac/pulumi/module/*.go` for actual cloud resources created
- `api.proto` for apiVersion and kind values

## Benefits

- **Developer-first catalog** — developers can find a working manifest in under 30 seconds
- **Source-verified accuracy** — every field, default, and resource claim traceable to source code
- **Scalable architecture** — catalog pages co-located with components, written by AI agents following strict rules
- **Gradual migration** — no disruption to existing catalog; components migrate one at a time
- **Audit capability** — dedicated audit rule ensures consistency across all catalog pages over time

## Impact

- 5 components now have production-quality catalog pages replacing research documents
- 173 components continue with legacy docs, ready for migration in future sessions
- Build pipeline supports both old and new documentation sources seamlessly
- Cursor rules enable any future session to write consistent catalog pages without re-establishing the standard
