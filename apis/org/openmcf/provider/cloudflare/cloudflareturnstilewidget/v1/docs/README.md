# CloudflareTurnstileWidget — Research & Design Notes

## What it is

A Turnstile widget is Cloudflare's CAPTCHA alternative. Creating one yields:

- a **site key** (public) embedded in the page's frontend widget, and
- a **secret key** used server-side to verify the token via the `/siteverify`
  endpoint.

Both are computed by the provider and surfaced as stack outputs.

## Single kind (no sub-resources)

A widget has no independently-composable sub-resources, so it is a single kind.

## Enums modeled as validated strings

`mode`, `clearance_level`, and `region` are fixed-value fields but are modeled as
plain strings with CEL `in [...]` validation rather than proto enums, because:

- `mode` includes the hyphenated value `non-interactive`, which is not a legal
  proto enum identifier; and
- `managed` is a valid value of BOTH `mode` and `clearance_level`. Proto enum
  value names are file-scoped, so two enums in this file could not both define
  `managed`.

Using the provider's exact strings keeps the spec faithful and avoids a lossy
enum→string mapping in the modules. The values are still a fixed, validated set
that a wizard can present as choices.

## Secret output

`secret` is exported as a sensitive output: Terraform marks the output
`sensitive = true`; Pulumi wraps it with `pulumi.ToSecret`. `sitekey` is public
and exported plainly. No spec field is secret-bearing (the secret is generated by
the provider), so there are no `(sensitive)` spec annotations.

## Immutability

`region` is `RequiresReplace` — it cannot change after creation.

## Engine parity

Optional flags (`clearance_level`, `bot_fight_mode`, `ephemeral_id`, `offlabel`,
`region`) are sent only when set (false booleans / empty strings become null /
omitted) so both engines fall back to the provider's defaults identically.

## Composition

- Outputs `sitekey` (frontend) and `secret` (server-side verification). A
  `CloudflareWorker` validating tokens references the `secret` as a managed-secret
  binding.
